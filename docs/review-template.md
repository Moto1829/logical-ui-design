# UIレビュー用テンプレート

このテンプレートは、画面レビューを好みではなく根拠で進めるための雛形です。レビューコメントを書くときは、問題、理由、影響、改善案の順で整理します。

レビューを始める前に、画面の目的、利用者、利用環境、変更制約をそろえておくと、感想ではなく判断として議論しやすくなります。

## 1. 画面の基本情報

- 画面名:
- URLまたは対象ファイル:
- 主目的:
- 主なユーザー行動:
- 主な利用者:
- 主な利用環境: デスクトップ / モバイル / 両方
- 利用頻度: 毎日 / ときどき / 初回中心
- 変更できない制約:

## 2. レビュー前にそろえること

- この画面で最初に見せるべき情報は何か
- この画面で最初に選ばせたい操作は何か
- 誤操作すると困る操作は何か
- 失敗時にユーザーが復帰できる必要があるか

ここが曖昧なままレビューすると、見た目の議論に寄りやすくなります。まず前提を合わせてから詳細を見ます。

## 3. 4つの軸で確認する

### すぐ理解できるか

- 画面の目的が一目で分かるか
- 重要情報が最初に見つかるか
- 用語や見出しが一貫しているか

### 迷わず選べるか

- 主操作が明確か
- 選択肢の違いが分かるか
- 情報量が多すぎないか

### 安心して操作できるか

- ボタン文言から結果を予測できるか
- 保存、送信、削除の違いが分かるか
- 処理中や完了後の状態が伝わるか

### 間違えても復帰できるか

- エラー時に回復方法が分かるか
- 破壊的操作に配慮があるか
- 戻る導線や再試行導線があるか

この4軸で全体を見たあと、必要に応じて画面タイプごとの観点を追加します。フォーム、一覧、詳細、モーダルでは見落としやすい点が少しずつ違います。

## 4. 画面タイプ別の補助チェック

### フォーム画面

- 必須項目と任意項目の違いが分かるか
- 入力順が利用者の思考順に合っているか
- エラーが修正支援になっているか
- 送信後に何が起きるか事前に分かるか

### 一覧画面

- 一目で比較すべき情報が比較しやすいか
- 並び順や絞り込み条件が分かるか
- 行クリックと行内操作が競合していないか
- 次の判断に必要な情報が不足していないか

### 詳細画面

- 対象と現在状態がすぐ分かるか
- 次に取りやすい行動が明確か
- 関連情報の優先順位が整理されているか
- 編集導線が必要以上に強くなっていないか

### モーダル

- 本当にモーダルである必要があるか
- 閉じ方と戻り先が明確か
- 背景にある必要情報を不自然に隠していないか
- 確認だけで済む内容に入力負荷を増やしていないか

## 5. アクセシビリティ確認

- 色だけに依存していないか
- コントラストは十分か
- フォーカス位置は見えるか
- ラベル、見出し、構造が明確か
- キーボードだけでも主要操作に到達できるか
- 状態変化が色以外でも理解できるか

## 6. コメント記入テンプレート

### 指摘テンプレート

- 問題:
- 理由:
- ユーザー影響:
- 改善案:
- 優先度: 高 / 中 / 低

### 記入例

- 問題: 主要ボタンと削除ボタンが近く、同じ強さで表示されている
- 理由: 役割の差が視覚的に伝わらず、誤操作しやすい
- ユーザー影響: 重要な操作の直前に不安が生じ、誤って削除するリスクがある
- 改善案: 削除ボタンを距離と色で分離し、主要ボタンのみ強く見せる
- 優先度: 高

### 書き出しに迷ったときの型

- 画面上のどこで問題が起きるか
- 想定ユーザーが何をしようとしているか
- そのとき何が分かりにくい、または危ないか
- どう変えると負荷が下がるか

### 曖昧な感想を具体化する例

- なんとなく使いにくい -> 初回利用者が完了までの流れを予測しにくい
- ちょっと見づらい -> 主見出しより補足情報が目立っていて優先順位が伝わりにくい
- ボタンが弱い気がする -> 主要操作が副次操作と同じ見た目で、どれを押すべきか迷いやすい

## 7. 優先度の付け方

- 高: 完了不能、誤操作、重大な不安、アクセシビリティ阻害につながる問題
- 中: 迷い、探索コスト、入力時間の増加につながる問題
- 低: 品質向上につながるが、影響は小さく、好みの差でも出やすい問題

優先度が並ぶときは、まず高だけを確定し、そのあと中と低を分けると判断しやすくなります。

## 8. レビュー会でのまとめ欄

- 高優先度の指摘:
- 追加確認が必要な指摘:
- 今回は見送る指摘:
- 担当者:
- 対応期限:

## 9. 最後に確認すること

- 修正案は画面の主目的を強めているか
- 細部ではなく根本原因に触れているか
- 特定のユーザーだけでなく、全体の使いやすさに寄与しているか
- 対応有無が決まっていない指摘が残っていないか
- コメントが単なる感想で終わっていないか
