# 第9章 テーブルとカードの使い分け

一覧を設計するとき、テーブルにするかカードにするかで迷うことがあります。これは見た目の好みの問題ではなく、ユーザーに何をさせたいかの問題です。ここでは、比較、探索、モバイル対応の観点から使い分けを整理します。

## 1. まず、何をしやすくしたいのかを決める

テーブルとカードは得意なことが違います。

- テーブル: 複数項目を横並びで比較しやすい
- カード: 1件ごとのまとまりを理解しやすい

そのため、先に「比較させたいのか」「1件ずつ把握させたいのか」を決める必要があります。

比較が中心ならテーブル、文脈理解が中心ならカード、という大枠があります。ここからは、それぞれが向く場面を具体的に見ます。

## 2. テーブルが向く場面

テーブルは、同じ列構造で複数データを比較したい場面に向いています。

例:

- 売上一覧
- 受注一覧
- ユーザー管理画面
- 在庫一覧

テーブルが向く理由:

- 同じ属性を縦横で比較しやすい
- 並び替えやフィルタとの相性が良い
- 情報密度を高く保ちやすい

ただし、列が増えすぎると横移動が増え、可読性が急激に落ちます。

## 3. カードが向く場面

カードは、1件ごとの状態や文脈をまとめて理解させたい場面に向いています。

例:

- 案件一覧
- 求人一覧
- 記事一覧
- プロジェクト一覧

カードが向く理由:

- 情報をまとまりとして把握しやすい
- 強調したい要素を柔軟に配置できる
- モバイルで崩しやすい

ただし、複数件の細かな比較には向きません。横断比較が必要な情報までカードにすると、比較コストが上がります。

## 4. 間違えやすい判断

よくある失敗は次の2つです。

- 比較したい情報をカードにしてしまう
- 1件ごとの文脈が大事なのにテーブルにしてしまう

前者では比較がしにくくなり、後者では一件ごとの意味が見えにくくなります。どちらも、見た目は整っていても、目的に対して不向きな設計です。

## 5. モバイルではどう考えるか

モバイルでは、テーブルはそのままでは苦しくなることが多いです。列数が多いと横スクロール頼みになり、探索しづらくなります。

対策の基本:

- 比較が本当に必要な列だけを残す
- 一部の列を折りたたむ
- モバイルではカードに変換する

一方で、カードはモバイルと相性が良いですが、件数が多いとスクロール量が増えます。どちらを選んでも、情報量と探索コストのバランスを見る必要があります。

ここまでの違いを踏まえると、最終的には画面の目的に照らして判断することが重要です。次は、迷ったときに使える判断基準を整理します。

## 6. 判断基準を持つ

迷ったときは、次の問いで判断すると整理しやすくなります。

- ユーザーは複数件を比較したいのか
- 1件ごとの背景や状態を見たいのか
- 主な操作は一覧上で行うのか、詳細へ入ってから行うのか
- モバイル利用はどれくらい多いのか

これに答えると、テーブルかカードか、または両方を使い分けるべきかが見えてきます。

一覧形式の選択を誤ると、ユーザーは本来簡単な比較や把握に余計な時間を使うことになります。見た目の整い方に反して、判断と探索の効率は大きく落ちます。

- 比較ミスが起きる: 比較向きの情報をカードにして、差分が見えなくなる
- 状態理解が浅くなる: 文脈が必要な情報を表だけに押し込み、1件ごとの意味が抜ける
- モバイルで破綻する: 横スクロールや長いカード列で探索コストが跳ね上がる
- 一覧離脱が増える: 何件も見ても判断しづらく、詳細画面へ過剰に頼るようになる

一覧形式は見た目の違い以上に、業務効率へ直結する設計判断です。最後に、この章の要点を振り返ります。

## 7. この章の要点

- テーブルとカードの違いは見た目ではなく目的で決める
- 比較が中心ならテーブル、まとまりの理解が中心ならカードが向く
- モバイルでは情報量と比較のしやすさを再設計する必要がある
- 迷ったら、比較か文脈理解かを基準に判断する
