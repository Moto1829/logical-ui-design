# 第13章 危険操作と安全設計

削除、退会、公開、上書き保存のような危険操作では、実行しやすさだけを追うと事故が起きます。一方で、すべてに重い確認ダイアログを付けると、日常操作まで遅くなります。ここでは、危険操作をどう見分け、どこで安全策を入れるべきかを整理します。

## 1. 危険操作とは何か

危険操作は、見た目が強い操作のことではありません。実行したあとに戻しにくい、影響範囲が広い、意図せず実行されると被害が大きい操作を指します。

代表例:

- データ削除
- アカウント退会
- 公開範囲の変更
- 一括更新
- 請求や送信の確定

つまり、危険性は色ではなく結果で判断します。

危険性を結果で判断できるようになると、次はどの場面で確認を入れるべきかを考えられます。ここからは、安全策の入れ方を整理します。

## 2. すべてに確認ダイアログを付ければ安全ではない

確認ダイアログは便利ですが、乱用すると読まれなくなります。毎回同じような確認を出すと、ユーザーは反射的に閉じるだけになります。

確認ダイアログが有効なのは次のような場合です。

- 取り消せない
- 影響範囲が大きい
- 実行意図の再確認が必要

逆に、後から取り消せる削除に毎回重い確認を挟むより、実行後に元に戻せる方が速くて安全なこともあります。

## 3. 誤操作を防ぐには配置と見た目を分ける

危険操作を安全にするには、文言だけでなく周辺配置も重要です。

見るべき点:

- 主要操作の近くに危険操作を置きすぎていないか
- 危険操作の色や形が通常操作と同じになっていないか
- ラベルが曖昧でないか

たとえば「保存」と「削除」が近くに並び、同じ強い色で表示されていると、視線だけで区別しにくくなります。

## 4. 戻せるなら戻せる設計を優先する

危険操作では、事前確認より事後回復の方が有効な場面があります。たとえば削除後に一定時間だけ元に戻せるなら、ユーザーは重い確認なしでも安心して操作できます。

考え方:

- 取り消せる操作: まず実行し、あとで戻せるようにする
- 取り消せない操作: 実行前に影響範囲を明示して確認する

重要なのは、どちらが安全かを操作コストと事故コストの両方で判断することです。

## 5. 確認文言は結果を具体的に書く

危険操作の確認文言で避けたいのは、抽象的な問いです。

悪い例:

- 本当に実行しますか
- 削除してもよろしいですか

良い例:

- この案件を削除すると、関連するメモと添付ファイルも削除されます。30日以内であれば復元できます
- 退会すると、請求履歴を除く個人設定は削除されます

何が起きるかが具体的であるほど、判断しやすくなります。

ここまでの観点を踏まえると、良い危険操作設計がどんな状態かをまとめやすくなります。次は、最低限満たしたい条件を整理します。

## 6. 良い危険操作設計の条件

良い危険操作設計は、次の問いに答えやすい状態です。

- これは本当に危険な操作か
- 実行すると何が起きるか
- 取り消せるか、取り消せないか

この3つが分かれば、ユーザーは必要以上に怖がらず、必要な場面では慎重になれます。

危険操作の設計が弱いと、ユーザーは二つの意味で損をします。誤って実行して被害を受けるか、逆に怖くて必要な操作までできなくなるかのどちらかです。

- 誤操作事故が起きる: 保存と削除の区別が弱く、意図しない実行につながる
- 判断が遅れる: 何が消えるか分からず、慎重になりすぎて作業が止まる
- 復旧負荷が高まる: 元に戻せない設計で、失敗の影響が長く残る
- サービスへの警戒感が強まる: 重要操作を安心して行えず、継続利用の心理的負担が増す

危険操作では、速さだけでも慎重さだけでも不十分です。最後に、この章の要点をまとめます。

## 7. この章の要点

- 危険操作は結果の重大さで見分ける
- 確認ダイアログは乱用せず、本当に必要な場面に限定する
- 配置、色、文言で誤操作を防ぐ
- 戻せる操作は事後回復を優先し、戻せない操作は影響範囲を具体的に示す
