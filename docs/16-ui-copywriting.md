# 第16章 UI文言設計

UIにおける文言は、装飾ではなく操作の一部です。どのボタンを押すのか、何が起きるのか、いま何を直せばよいのかは、見た目だけでは伝わりません。ここでは、理解しやすく誤解しにくいUI文言の基本を整理します。

## 1. ボタン文言は「押した結果」を表す

ボタンの文言は、見た目の短さより、押したあとに何が起きるかが分かることが重要です。

悪い例:

- 実行
- OK
- 送信

良い例:

- 案件を保存する
- 請求書を送信する
- 変更を公開する

抽象的な言葉ほど、確認コストが高くなります。ユーザーは前後の文脈を読み直さないと判断できません。

## 2. 補助説明は不安を減らすために書く

補助説明は、すべてを説明する欄ではありません。ユーザーが迷いやすい点、不安になりやすい点だけを短く支えるために使います。

書くと効果がある内容:

- 入力形式
- 実行後の影響
- 後から変更できるかどうか

長すぎる補助説明は読まれにくく、主要情報を埋もれさせます。

## 3. エラー文言は原因と修正方法を返す

エラー文言では、起きたことだけでなく、どう直せばよいかまで返す必要があります。

悪い例:

- 入力が正しくありません
- エラーが発生しました

良い例:

- メールアドレスの形式で入力してください
- 8文字以上でパスワードを入力してください

「何が悪いか」ではなく、「どう直せば進めるか」に焦点を置くと、行き止まりを減らせます。

## 4. 空状態と完了状態では感情の支え方が違う

空状態では次の一歩を示す必要があり、完了状態では安心して次へ進める必要があります。同じ調子の文言では不十分です。

例:

- 空状態: まだ案件がありません。最初の案件を追加すると一覧で管理できます
- 完了状態: 案件を保存しました。続けて担当者を設定できます

状態ごとに、必要な支援は変わります。

## 5. ユーザーを責めない言い方にする

文言で避けたいのは、ユーザーの失敗を断定的に責める表現です。UIは案内役なので、非難ではなく支援の言葉を返すべきです。

避けたい例:

- 正しく入力してください
- 不正な操作です
- エラーのため失敗しました

より良い例:

- 日付は YYYY/MM/DD 形式で入力してください
- この操作には管理者権限が必要です
- 通信が不安定なため保存できませんでした

## 6. 良いUI文言の条件

良いUI文言は、次の問いに答えやすい状態です。

- 押すと何が起きるか
- いま何が起きているか
- 次に何をすればよいか

この3つが文言で支えられていれば、画面理解は大きく改善します。

文言が曖昧なUIでは、見た目が整っていても判断の責任がユーザー側に押し戻されます。つまり、読むたびに意味を補完しなければならず、小さな迷いが積み重なります。

- 押下前の不安が増える: ボタン結果が読めず、操作をためらう
- 修正が進まない: エラー文言が抽象的で、どこを直すか分からない
- 状態を誤解する: 完了や空状態の意味が弱く、次の行動を決めにくい
- 学習コストが上がる: 同じ画面でも毎回文脈を読み直す必要がある

## 7. この章の要点

- ボタン文言は抽象語ではなく結果で書く
- 補助説明は不安と迷いを減らすために短く添える
- エラー文言は原因と修正方法を返す
- 状態に応じて文言の役割を変える
