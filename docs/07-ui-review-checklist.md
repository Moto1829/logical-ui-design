# 第7章 UIレビューの観点

UIレビューで重要なのは、好みの表明ではなく、問題を説明できることです。この章では、レビュー時に使える観点を整理します。

## 1. まず確認する4つの軸

- すぐ理解できるか
- 迷わず選べるか
- 安心して操作できるか
- 間違えても復帰できるか

この4軸は、どの画面にも共通して使える基準です。レビュー時に観点が散らばるときは、まずこの4つに戻ると整理しやすくなります。

たとえば「見た目がごちゃついている」という感想も、分解すると次のように言い換えられます。

- 情報の優先順位が分からず、すぐ理解できない
- 主要ボタンが埋もれていて、迷わず選べない
- 破壊的操作が近く、不安なく操作できない

## 2. 画面レビューのチェック項目

### 情報設計

- 最も重要な情報が最初に目に入るか
- 見出しと本文の関係が明確か
- 補足情報が主情報を邪魔していないか

補足で確認したい点:

- 画面タイトルだけで目的が伝わるか
- 一つの画面で複数の主目的が競合していないか
- 重要な情報が折りたたみの中に隠れすぎていないか

### 操作性

- 主要ボタンが明確か
- タップやクリックの対象が十分な大きさか
- 危険な操作が通常操作と区別されているか

補足で確認したい点:

- 連打や二重送信が起きにくいか
- モバイルで親指操作しやすい位置か
- 戻る、キャンセル、削除の違いが明確か

### 理解しやすさ

- 用語が一貫しているか
- ボタン文言が結果を予測できるか
- エラーや補助説明が具体的か

補足で確認したい点:

- 専門用語や社内用語がそのまま出ていないか
- 省略しすぎて意味が曖昧になっていないか
- 空状態や初回状態で次の行動が分かるか

### アクセシビリティ

- 色だけに依存していないか
- コントラストが不足していないか
- キーボードやスクリーンリーダー利用を妨げないか

補足で確認したい点:

- フォーカス位置が視認できるか
- 見出し構造が自然か
- フォームのラベルと説明が関連付いているか

## 3. 画面タイプ別の見方

すべての画面を同じ観点で見てもよいですが、画面タイプごとに重視点は変わります。

### フォーム画面

- 入力順が自然か
- 途中で不安にならないか
- エラーが修正支援になっているか

### 一覧画面

- 一目で比較できるか
- 行動に必要な情報が不足していないか
- フィルタや検索が探索を助けているか

### モーダル

- 本当にモーダルである必要があるか
- 閉じ方が明確か
- 元の文脈へ戻りやすいか

## 4. レビューコメントの書き方

良いレビューコメントは、次の形で書けます。

1. どこに問題があるか
2. なぜ問題なのか
3. ユーザーにどんな負荷があるか
4. どう直すとよいか

この順番が重要なのは、単なる好みや感想で終わらせないためです。改善案だけを先に言うと、なぜ変える必要があるのかが伝わりにくくなります。

例:

「送信ボタンの近くに削除ボタンが同じ強さで配置されているため、誤操作のリスクがあります。主要操作と破壊的操作の視覚差を強め、距離も分けた方が安全です。」

別の例:

「検索条件が画面上部に詰め込まれており、初回利用時にどこから見ればよいか分かりにくいです。まず一覧の目的と主要フィルタを分けて見せると、探索コストを下げられます。」

## 5. レビューで避けたい言い方

次のような表現は、意図が曖昧で改善につながりにくいです。

- なんとなく使いにくい
- ちょっと見づらい
- もっと今っぽくしたい
- ボタンが弱い気がする

こうした感想が出たときは、次のように言い換えます。

- 何が見つけにくいのか
- 何と何の区別がつきにくいのか
- どの操作が不安なのか
- どのユーザー行動が止まりそうなのか

## 6. 実務で使うレビューの進め方

レビューを効率よく進めるには、次の順で見ると判断がぶれにくくなります。

1. 画面の目的を確認する
2. 主情報と主操作を特定する
3. 認知負荷が高い箇所を洗い出す
4. 誤操作や不安につながる箇所を確認する
5. 最後に文言や細部を詰める

装飾や細かな余白から入ると、根本の問題を見落としやすくなります。まず構造、次に操作、最後にディテールという順が実務では有効です。

## 7. この章の要点

- UIレビューは好みではなく根拠で行う
- 情報設計、操作性、理解しやすさ、アクセシビリティで分けて見る
- 問題、理由、影響、改善案の順にコメントすると伝わりやすい
