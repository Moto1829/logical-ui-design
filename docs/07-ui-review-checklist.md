# 第7章 UIレビューの観点

UIレビューで重要なのは、好みの表明ではなく、問題を説明できることです。この章では、レビュー時に使える観点を整理します。

## 1. まず確認する4つの軸

- すぐ理解できるか
- 迷わず選べるか
- 安心して操作できるか
- 間違えても復帰できるか

この4軸は、どの画面にも共通して使える基準です。レビュー時に観点が散らばるときは、まずこの4つに戻ると整理しやすくなります。

たとえば「見た目がごちゃついている」という感想も、分解すると次のように言い換えられます。

- 情報の優先順位が分からず、すぐ理解できない
- 主要ボタンが埋もれていて、迷わず選べない
- 破壊的操作が近く、不安なく操作できない

この4軸で違和感の所在をつかんだら、次はどの観点で問題を分解するかを見ます。ここから先は、レビュー時に見落としを減らすための具体的なチェック項目です。

## 2. 画面レビューのチェック項目

4つの軸は判断の土台ですが、実際のレビューではもう少し細かく問題を切り分ける必要があります。そこで、画面を情報設計、操作性、理解しやすさ、アクセシビリティの4つに分けて確認します。

### 情報設計

- 最も重要な情報が最初に目に入るか
- 見出しと本文の関係が明確か
- 補足情報が主情報を邪魔していないか

補足で確認したい点:

- 画面タイトルだけで目的が伝わるか
- 一つの画面で複数の主目的が競合していないか
- 重要な情報が折りたたみの中に隠れすぎていないか

### 操作性

- 主要ボタンが明確か
- タップやクリックの対象が十分な大きさか
- 危険な操作が通常操作と区別されているか

補足で確認したい点:

- 連打や二重送信が起きにくいか
- モバイルで親指操作しやすい位置か
- 戻る、キャンセル、削除の違いが明確か

### 理解しやすさ

- 用語が一貫しているか
- ボタン文言が結果を予測できるか
- エラーや補助説明が具体的か

補足で確認したい点:

- 専門用語や社内用語がそのまま出ていないか
- 省略しすぎて意味が曖昧になっていないか
- 空状態や初回状態で次の行動が分かるか

### アクセシビリティ

- 色だけに依存していないか
- コントラストが不足していないか
- キーボードやスクリーンリーダー利用を妨げないか

補足で確認したい点:

- フォーカス位置が視認できるか
- 見出し構造が自然か
- フォームのラベルと説明が関連付いているか

ここまでが画面を横断して使える基本観点です。ただし、同じ観点でも画面タイプによって見るべき箇所は変わります。次は、実務で頻出する画面ごとの見方に落とします。

## 3. 画面タイプ別の見方

すべての画面を同じ観点で見てもよいですが、画面タイプごとに重視点は変わります。

### フォーム画面

- 入力順が自然か
- 途中で不安にならないか
- エラーが修正支援になっているか

実務で見落としやすい点:

- 必須項目と任意項目の区別が一目で分かるか
- 入力途中で保存できるべき内容なのに、やり直し前提になっていないか
- 確認画面や完了画面がなくても、結果が十分に伝わるか

### 一覧画面

- 一目で比較できるか
- 行動に必要な情報が不足していないか
- フィルタや検索が探索を助けているか

実務で見落としやすい点:

- 並び順の基準が分かるか
- 件数が多いときに絞り込み条件の状態が把握できるか
- 行クリックと行内ボタンが競合して誤操作を生まないか

### 詳細画面

- その対象が何で、今どんな状態かがすぐ分かるか
- 次に取りやすい行動が明確か
- 関連情報が散らばりすぎていないか

実務で見落としやすい点:

- ステータス、更新日時、担当者など判断に必要な情報が上部で把握できるか
- 閲覧中心の画面なのに、編集導線が強すぎないか
- 履歴、関連データ、添付情報の優先順位が整理されているか

### モーダル

- 本当にモーダルである必要があるか
- 閉じ方が明確か
- 元の文脈へ戻りやすいか

実務で見落としやすい点:

- 背景を隠すことで必要な参照情報まで失っていないか
- 確認だけで済む内容なのに、入力や判断を増やしていないか
- 閉じたあとに入力内容やスクロール位置が不自然に失われないか

画面タイプごとに重点を変えると、レビューコメントはかなり具体的になります。次は、その観点をどう文章に落とすかを整理します。

## 4. レビューコメントの書き方

良いレビューコメントは、次の形で書けます。

1. どこに問題があるか
2. なぜ問題なのか
3. ユーザーにどんな負荷があるか
4. どう直すとよいか

この順番が重要なのは、単なる好みや感想で終わらせないためです。改善案だけを先に言うと、なぜ変える必要があるのかが伝わりにくくなります。

実務では、1件のコメントを長く書くより、短くてもこの順番を守る方が有効です。特に「誰が」「どの場面で」「どう困るか」が入ると、優先度の判断がしやすくなります。

書き出しに迷うときは、次の型を使えます。

- 画面上のどこで問題が起きるか
- 想定ユーザーが何をしようとしているか
- そのとき何が分かりにくい、または危ないか
- どう変えると負荷が下がるか

例:

「送信ボタンの近くに削除ボタンが同じ強さで配置されているため、誤操作のリスクがあります。主要操作と破壊的操作の視覚差を強め、距離も分けた方が安全です。」

別の例:

「検索条件が画面上部に詰め込まれており、初回利用時にどこから見ればよいか分かりにくいです。まず一覧の目的と主要フィルタを分けて見せると、探索コストを下げられます。」

もう一つの例:

「承認依頼のステータスがラベル色だけで表現されているため、色覚差がある利用者には区別しにくい可能性があります。ラベル文言やアイコンも併記して、状態を色以外でも判別できるようにした方が安全です。」

レビューコメントは、次の節のように曖昧な言い方を避けることで、さらに実務で使いやすくなります。

## 5. レビューで避けたい言い方

次のような表現は、意図が曖昧で改善につながりにくいです。

- なんとなく使いにくい
- ちょっと見づらい
- もっと今っぽくしたい
- ボタンが弱い気がする

こうした感想が出たときは、次のように言い換えます。

- 何が見つけにくいのか
- 何と何の区別がつきにくいのか
- どの操作が不安なのか
- どのユーザー行動が止まりそうなのか

実務では、曖昧な感想をその場で否定する必要はありません。まず感想を受け止めてから、観点を足して具体化すると議論が進みやすくなります。

- 「なんとなく使いにくい」ではなく、「初回利用者が完了までの流れを予測しにくい」
- 「ちょっと見づらい」ではなく、「主見出しより補足情報が目立っていて優先順位が伝わりにくい」
- 「ボタンが弱い気がする」ではなく、「主要操作が副次操作と同じ見た目で、どれを押すべきか迷いやすい」

ここまでで、何を見るかとどう書くかがそろいました。最後に、実務のレビューを安定して回すための進め方を整理します。

## 6. 実務で使うレビューの進め方

レビューを効率よく進めるには、次の順で見ると判断がぶれにくくなります。

1. 画面の目的を確認する
2. 主情報と主操作を特定する
3. 認知負荷が高い箇所を洗い出す
4. 誤操作や不安につながる箇所を確認する
5. 最後に文言や細部を詰める

装飾や細かな余白から入ると、根本の問題を見落としやすくなります。まず構造、次に操作、最後にディテールという順が実務では有効です。

### レビュー前にそろえる情報

- この画面の目的は何か
- 主な利用者は誰か
- どの環境でよく使われるか
- 変更できない制約はあるか

これらが曖昧なままレビューを始めると、良し悪しではなく前提の食い違いで議論が長引きます。特に業務画面では、利用頻度、入力件数、モバイル利用の有無だけでも事前に共有しておくと判断がかなり安定します。

### 指摘の優先度をそろえる

- 高: 完了不能、誤操作、重大な不安、アクセシビリティ阻害につながる
- 中: 迷い、探索コスト、入力時間の増加につながる
- 低: 好みの差で済む表現や、影響の小さい見た目の粗さ

このように優先度の基準を合わせておくと、コメント数が多くても何から直すべきか判断しやすくなります。レビュー会で全件を同じ重さで扱うと、重大な問題が埋もれやすくなります。

### レビュー会での進め方

- 最初に画面の目的と対象ユーザーを確認する
- 1人ずつ話す前に、数分だけ各自で静かに確認する
- 指摘は「問題」「影響」「優先度」の順で共有する
- その場で結論が出ないものは、追加確認事項を残す
- 最後に対応有無と担当を決める

レビューは、良い案をその場で出し切ることより、問題の認識をそろえることの方が重要です。原因の整理が十分なら、改善案はあとからでも質を上げやすくなります。

また、レビューで悪影響を言語化できないと、画面の問題は「好みの違い」として処理されやすくなります。その結果、ユーザーが実際に受けている負荷が放置され、同じ失敗が繰り返されます。

- 問題が軽視される: 影響が説明されないため、修正優先度が上がらない
- 改善がぶれる: 担当者ごとの感覚で直し方が変わり、品質が安定しない
- ユーザー負荷が残る: 迷い、誤操作、不安の原因が構造的に残り続ける
- 学習が蓄積しない: チーム内で再利用できるレビュー知見にならない

この章で扱った観点は、単発のレビューだけでなく、チームの共通言語として使うと効果が出ます。最後に、実務で最低限押さえたい要点をまとめます。

## 7. この章の要点

- UIレビューは好みではなく根拠で行う
- 情報設計、操作性、理解しやすさ、アクセシビリティで分けて見る
- 問題、理由、影響、改善案の順にコメントすると伝わりやすい
