# 第3章 情報設計とレイアウト

UIが分かりにくい原因は、個々のパーツではなく、画面全体の情報設計にあることが少なくありません。ここでは、情報の優先順位と配置をどう考えるかを整理します。

## 1. まず優先順位を決める

すべてを強調すると、何も強調されていないのと同じです。画面内の情報は少なくとも次の3段階に分けて考える必要があります。

- 最初に見せるべき情報
- 読めば理解が深まる補足情報
- 必要な人だけが見る詳細情報

この整理をせずにUIを作ると、見出しも説明も警告もボタンもすべてが同じ強さになり、結果としてユーザーは自分で優先順位を判断しなければなりません。

設計時には、各要素に対して「最初の3秒で見えていなければ困るか」を問いかけると優先順位が見えやすくなります。

## 2. レイアウトは視線を制御する

レイアウトは見た目を整えるためではなく、視線の順番を制御するためにあります。見出し、本文、補助情報、操作の並びが自然であれば、ユーザーは迷いにくくなります。

重要な観点:

- 左上から右下への流れを妨げていないか
- 重要情報が孤立していないか
- 余白が意味の区切りとして機能しているか

たとえば設定画面で、説明文が左側、スイッチが離れた右端、保存ボタンがさらに下部にあると、視線が何度も往復します。これは単なる見た目の散らかりではなく、操作までの理解コストの上昇です。

良いレイアウトは、読み順と操作順が自然につながります。見出しを読み、選択肢を確認し、最後に行動できるという流れが壊れていないかを確認します。

## 3. 余白は装飾ではなく構造

余白が不足すると、要素同士の関係が読み取りにくくなります。逆に適切な余白は、見出しと本文、ラベルと入力欄、本文とボタンの関係を自然に伝えます。

余白の設計では、すべてを均等にする必要はありません。むしろ、関係が強い要素は近く、関係が弱い要素は離すことで構造を伝えます。

見るべき点:

- セクション間の余白は十分か
- ラベルと入力欄の距離は近すぎず遠すぎないか
- 主要ボタンの前に、判断のための間があるか

## 4. タイポグラフィは可読性の設計

文字サイズ、行間、太さ、行長は、すべて理解速度に影響します。

悪い例:

- 重要見出しと本文の差が小さい
- 一行が長すぎて視線が戻りにくい
- 補助情報が薄すぎて読めない

良い例:

- 見出しと本文で役割の差が明確である
- 行間に十分な余裕がある
- 補助情報も読めるが主張しすぎない

さらに重要なのは、一つの画面で文字のルールが安定していることです。見出し、本文、注釈、ボタンラベルが毎回違うルールで組まれていると、読み手は意味の差ではなく見た目の差を都度解釈しなければなりません。

## 5. 色は意味を補強するために使う

色は雰囲気づくりだけでなく、状態や優先度を伝えるために使います。ただし色だけに意味を持たせると、判別できないユーザーが出ます。

そのため、色に加えて次の要素を併用します。

- ラベル
- アイコン
- 位置
- 形状の違い

また、ブランドカラーを強く使う場面は絞る方がよいです。画面内のすべてが鮮やかな色だと、重要な状態変化や主要ボタンの強調に使える余地がなくなります。

## 6. レイアウトレビューの実践

画面全体を見るときは、次の順番で確認すると判断しやすくなります。

1. 画面の目的が最初に分かるか
2. 主情報と補助情報が分離されているか
3. 主要操作が視線の終点にあるか
4. 危険操作が十分に離れているか

これはデザインツール上でも、実装後の画面でも同じです。視線の流れと情報の階層が崩れていれば、装飾を足しても根本解決にはなりません。

情報設計とレイアウトが悪いと、ユーザーは内容を理解する前に「どこから見ればよいか」を自分で決めなければなりません。これは画面理解の失敗を増やし、重要情報の見落としにつながります。

- 優先順位を誤る: 警告や主要CTAが埋もれ、重要度の低い情報から読んでしまう
- 視線が往復する: 説明、操作、補助情報が離れすぎており、理解と操作が分断される
- 読み飛ばしが増える: 構造が弱いため、必要な情報を見落とす
- 疲労感が強まる: 画面を使うたびに整理し直す必要があり、継続利用で負担が蓄積する

## 7. この章の要点

- 情報設計では、何を先に見せるかを明確にする
- レイアウトと余白は意味の区切りを伝えるために使う
- 文字と色は感覚ではなく可読性と意味伝達のために設計する
