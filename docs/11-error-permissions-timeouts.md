# 第11章 エラー、権限不足、タイムアウトの設計

失敗時のUIは、単に問題を通知するためのものではありません。ユーザーが状況を理解し、次にどう行動すればよいかを示す必要があります。ここでは、エラー、権限不足、タイムアウトといった失敗状態をどう設計するかを整理します。

## 1. 失敗状態にも種類がある

「エラー」と一言でまとめると、必要な支援が曖昧になります。少なくとも次の違いを分けて考えるべきです。

- 入力不備: ユーザーがその場で修正できる
- 権限不足: 自分では解決できない可能性がある
- 通信失敗やタイムアウト: 再試行で回復できる可能性がある
- システム障害: 待つ、問い合わせるなど別の行動が必要

種類が違えば、表示すべき説明も導線も変わります。

まずは種類を分けることが出発点ですが、それだけでは回復を助けられません。次に、ユーザーが行動を決められるだけの説明の出し方を見ます。

## 2. 何が起きたかを曖昧にしない

「エラーが発生しました」は、ほとんど何も伝えていません。重要なのは、ユーザーが取るべき行動を決められるだけの情報を返すことです。

最低限必要な要素:

- 何が起きたか
- 自分で直せるか
- 次に何をすればよいか

悪い例:

- エラーが発生しました

良い例:

- 接続が不安定なため保存できませんでした。通信環境を確認して再試行してください

## 3. 権限不足は責任の所在を明確にする

権限不足は、ユーザーの操作ミスではないことが多いです。それにもかかわらず「アクセスできません」だけだと、何が不足しているのか分からず、行き止まりになります。

権限不足で必要な情報:

- 何にアクセスできないのか
- なぜできないのか
- 誰に依頼すればよいか

例:

- このレポートは管理者のみ閲覧できます。必要な場合は管理者へ閲覧権限を依頼してください

## 4. タイムアウトでは再試行の道を残す

タイムアウトや一時的な通信失敗は、再試行で回復することがあります。そのため、ユーザーを行き止まりにしないことが重要です。

見るべき点:

- いま再試行すればよいのか
- 入力内容は保持されているのか
- 他の操作をすべきなのか

特にフォームでは、失敗時に入力内容が消えると、失敗そのものより大きなストレスになります。

## 5. 失敗時にユーザーを責めない

失敗文言で避けたいのは、ユーザーに責任を押し付ける表現です。ユーザーが悪いのではなく、いま起きている状態を一緒に解決する姿勢が必要です。

避けたい表現:

- 正しく入力してください
- 操作に失敗しました
- 不正な操作です

より良い表現:

- メールアドレスの形式で入力してください
- 接続が不安定なため保存できませんでした
- この操作には管理者権限が必要です

こうした文言の違いが積み重なると、失敗状態は混乱ではなく回復の案内になります。次は、良い失敗状態が満たすべき条件を整理します。

## 6. 良い失敗状態の条件

良い失敗状態は、次の問いに答えやすい状態です。

- 何が起きたか
- いま自分で解決できるか
- 次に何をすればよいか

この3つが分かれば、失敗状態は混乱の原因ではなく、回復の起点になります。

失敗状態の案内が弱いUIでは、ユーザーは失敗そのものより、その後どうすればよいか分からないことに強く消耗します。作業中断や再入力、問い合わせ増加にもつながります。

- 回復できずに止まる: 再試行すべきか別手段を取るべきか判断できない
- 自責感が生まれる: 曖昧なエラー文言で、自分の操作が悪いと思い込む
- 再入力負荷が増える: 入力保持がなく、失敗のたびに最初からやり直す
- サポート依存が増える: 権限不足や障害の切り分けができず、問い合わせが必要になる

失敗時のUIは、問題の通知ではなく回復支援として設計する必要があります。最後に、この章の要点をまとめます。

## 7. この章の要点

- 失敗状態は種類ごとに分けて設計する
- 何が起きたか、どう回復するかを具体的に示す
- 権限不足とタイムアウトでは必要な支援が違う
- ユーザーを責めず、次の行動を示す
