# 第5章 状態変化とフィードバック

UIは、押せることだけでなく、押した結果が伝わることまで含めて設計する必要があります。操作後の変化が曖昧だと、ユーザーは不安になり、同じ操作を繰り返したり、処理失敗だと誤解したりします。

## 1. フィードバックが必要な場面

特に次の場面では、明確なフィードバックが必要です。

- ボタンを押した直後
- 通信中や保存中
- 完了したとき
- 失敗したとき
- 元に戻せない操作をしたとき

逆に言えば、これらの場面で何も返ってこないUIは、ユーザーに「今どうなっているのか」を推測させています。推測が必要なUIは、それだけで不安を増やします。

## 2. システム状態は見えるようにする

ユーザーは今何が起きているかを理解したいと考えます。読み込み中、保存済み、未保存、失敗、再試行可能など、状態を表現しないUIは不親切です。

特に通信が発生する操作では、反応がない時間が短くても「押せていないのではないか」という疑念が生まれます。ボタンのローディング表示、進行中メッセージ、再試行導線は、その疑念を減らすための設計です。

## 3. 完了メッセージは次の行動につなげる

単に「保存しました」と表示するだけでは不十分なことがあります。完了後に何をしてほしいのかを示すと、体験が途切れません。

例:

- 保存しました。設定一覧に戻る
- 登録が完了しました。プロフィールを設定する
- 送信しました。内容を編集する

良い完了メッセージは、単なる報告ではなく、文脈の継続です。いま何が完了し、次に何ができるかまで含めて伝えると、操作が途切れません。

## 4. 失敗時は責任を押し付けない

「エラーが発生しました」だけでは、ユーザーは何もできません。少なくとも次を示す必要があります。

- 何が起きたか
- どうすれば回復できるか
- 再試行できるか

ユーザーの入力ミスなのか、通信不良なのか、システム障害なのかで、必要な支援は変わります。すべてを同じ「エラーが発生しました」で片づけると、適切な行動が取れません。

悪い例:

- 保存に失敗しました

良い例:

- 通信が不安定なため保存できませんでした。接続を確認してもう一度お試しください
- メールアドレスが重複しています。別のアドレスで登録してください

## 5. フィードバックの強さを揃えすぎない

成功、注意、失敗、処理中のすべてが同じ見た目だと、重要度が伝わりません。逆に、すべてを強い色で出すと、どれも重大に見えてしまいます。

設計のポイント:

- 成功は安心感を与えるが主張しすぎない
- 注意は次の判断に必要な情報として目立たせる
- 失敗は原因と回復手段が最優先で伝わるようにする
- 処理中は待てばよいのか、操作が必要なのかを分ける

フィードバックが弱いUIでは、ユーザーは「押せたのか」「壊れたのか」「待てばよいのか」を推測で補うしかありません。その結果、二重送信や不必要な離脱、サービス不信が起きます。

- 二重操作が増える: 反応が見えず、同じボタンを繰り返し押してしまう
- 処理失敗と誤解する: 単なる待機中でも、保存されていないと思って離脱する
- 失敗時に止まる: 回復方法が見えず、作業継続を諦める
- 不安が蓄積する: 毎回結果が読めない体験が続くと、安心して使えなくなる

## 6. この章の要点

- 操作可能性だけでなく、状態の可視化まで設計する
- 読み込み、完了、失敗の各状態を明確に伝える
- フィードバックは不安を減らし、次の行動を支える
