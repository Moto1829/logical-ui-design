# 第17章 UIレビューのケーススタディ

UIレビューで難しいのは、違和感を感じることではなく、それを改善可能な言葉に落とすことです。この章では、架空の画面を題材に、問題点の見つけ方とレビューコメントの書き方を実例で整理します。

## 1. レビューでは好みより根拠を優先する

「なんとなく見づらい」「もっと整えた方がよい」といった感想だけでは、改善につながりません。レビューでは、問題を次の観点に分解すると扱いやすくなります。

- 情報の優先順位が崩れている
- 操作の意味が分かりにくい
- 状態変化や結果が見えにくい
- アクセシビリティや可読性に問題がある

感想を構造に変換すると、議論が前に進みます。

## 2. ケーススタディの前提を明確にする

レビューでは、誰が何のために使う画面かが曖昧だと評価がぶれます。たとえば、営業担当が毎日使う案件一覧と、管理者が月1回見る設定画面では、重要な観点が違います。

今回のケースでは、次の前提を置きます。

- 画面: 案件一覧
- 利用者: 営業担当
- 主目的: 優先案件の確認と次の対応開始

この前提があると、何が主要で何がノイズかを判断しやすくなります。

## 3. 問題点を画面上の事実として拾う

レビューでは、まず画面に起きている事実を観察します。

例:

- 強い色の要素が多く、優先順位が分からない
- ボタン文言が抽象的で、結果が読み取れない
- 検索条件と結果件数の関係が分からない
- 期限切れ案件が通常案件に埋もれている

この段階では、好き嫌いではなく観察結果を書くことが重要です。

## 4. コメントは問題と改善案をセットにする

良いレビューコメントは、問題指摘だけで終わりません。なぜ問題なのか、どう直せるかまで含めます。

悪い例:

- なんとなく見づらいです
- ボタンをもっと分かりやすくしてください

良い例:

- 期限切れ案件と通常案件の見た目の差が小さく、優先順位が判断しにくいです。期限切れだけを上段に分離するか、強いラベルを付けると判断しやすくなります
- 「実行」ボタンでは結果が分かりません。案件を開始する操作なら「対応を開始する」のように結果が読める文言が適切です

## 5. 改善後に何が良くなったかも言語化する

レビューは欠点探しだけではありません。改善後に何がどう良くなったかを言葉にすると、チームの判断基準が蓄積されます。

見るべき変化:

- 視線の流れが整理されたか
- 次の操作が早く選べるか
- 状態の違いが誤読されにくくなったか

改善の理由が共有されるほど、次の画面でも品質が安定します。

## 6. 良いレビューの条件

良いレビューは、次の問いに答えやすい状態です。

- 何が問題なのか
- なぜそれが問題なのか
- どう直せばよいか

この3つが揃っていれば、レビューは個人の感想ではなく、再利用できる知見になります。

レビューケースで見えるように、悪いUIの影響は単発の使いにくさでは終わりません。優先順位の誤読、行動開始の遅れ、不必要な確認、誤操作不安が連鎖し、業務全体の速度と安心感を下げます。

- 優先案件を見落とす: 見た目の差が弱く、重要状態に気づくのが遅れる
- 操作開始が遅れる: ボタン文言や配置が曖昧で、次の行動を決めにくい
- 余計な確認が増える: 一覧だけで判断できず、詳細や他者確認に頼る
- 同じ問題が再発する: レビューで影響を言語化しないと、別画面でも類似不備が残る

## 7. この章の要点

- レビューでは好みより根拠を優先する
- 画面の前提と目的を明確にして評価する
- 問題指摘は観察事実として書く
- コメントは問題、理由、改善案をセットにする
